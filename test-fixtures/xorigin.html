Cross-origin re-fetch — permission prompt & SSRF deny

The main fixtures (index.html) serve every file same-origin, so the content script reads the bytes itself and the background never re-fetches. This page links to files on other origins, which exercises the background's credentialed re-fetch — the path guarded against SSRF in issue #23.

Setup
  1. Add two throwaway hostnames that resolve to your machine (removed in the Cleanup step): 
    # /etc/hosts  (sudo)
127.0.0.1  ctm-page.test  ctm-files.test
    No-sudo alternative: skip this and use the files.lvh.me link below — *.lvh.me already resolves to 127.0.0.1 via public DNS (may be blocked on some networks).
  2. npm run build, then load dist/chrome unpacked.
  3. Sign in to Color The Map (toolbar icon → Connect) with at least one map.
  4. npm run fixtures (serves this folder on port 8080).
  5. Open this page at http://ctm-page.test:8080/xorigin.html (or http://localhost:8080/xorigin.html).

1. Cross-origin public host — expect a permission prompt

These point at valid.gpx on a different hostname than the page. The host isn't a private/loopback address, so the extension treats it like any normal site and asks the browser for access before re-fetching.

Expect: click → a new tab opens the file and the Send to Color The Map toast appears here → pick a map → Send → the browser shows “Allow this extension to access ctm-files.test?” → Allow → the track imports (“You're on the map”). The toast also shows “from ctm-files.test” as the source.

  • valid.gpx from ctm-files.test — needs the /etc/hosts line above.
  • valid.gpx from files.lvh.me — no /etc/hosts needed (public DNS → 127.0.0.1).

2. Internal / loopback targets — expect a block (no prompt, no fetch)

These point at loopback and private-network addresses — the kind of target a malicious page would use to make the extension probe your own machine or LAN. The SSRF guard refuses them before any request.

Expect: click → toast appears → Send → no permission prompt, and a “couldn't be imported” card. Confirm nothing was fetched: chrome://extensions → this extension → service worker → DevTools → Network → there is no request to the address below. (The 127.0.0.1 file really exists on the server, so the block is genuine — not just a dead link.)

  • valid.gpx from 127.0.0.1 (loopback)
  • 169.254.169.254 (cloud metadata, link-local)
  • 192.168.1.1 (typical router, private)

3. Opt-in — “Dangerous features” toggle

Open the extension's Options page (right-click the toolbar icon → Options, or chrome://extensions → Details → Extension options) and enable “Allow importing from private network addresses.”

Expect: retry the 127.0.0.1 link from section 2 → now Send prompts for 127.0.0.1 and, on Allow, actually imports. Turn the toggle back off when done.

4. Extras worth a look

  • Closed shadow root: with a toast open, run document.querySelector('ctm-upload-toast').shadowRoot in this page's console — it returns null (the page can't read the toast).
  • Source host: every toast names where the file would be fetched from (“from …”).

5. Cleanup

  • Stop npm run fixtures.
  • Remove the ctm-page.test ctm-files.test line from /etc/hosts.
  • Optionally revoke the granted host permission: chrome://extensions → this extension → Details → Site access.
